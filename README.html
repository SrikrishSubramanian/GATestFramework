GATestFramework Playwright E2E Test Suite for Global Atlantic (AEM)

This guide covers everything you need to install, run, and review automated tests for Global Atlantic web components. No coding experience required — just follow the commands below.

What does this framework test? UI rendering, accessibility (WCAG 2.2), visual regression, broken images/links, responsive layout, and component interactions — across Chrome, Safari, and mobile devices.

Installation

  1. Install Node.js
    Download and install Node.js (v14 or later). This includes npm.

    PlatformHow to install
    Windows Download the .msi installer from nodejs.org and run it. Defaults are fine.
    Mac Download the .pkg installer from nodejs.org, or use Homebrew: brew install node
    Verify the install by opening a terminal and running: 
    node --version
npm --version
  2. Open a terminal
    PlatformWhich terminal to use
    Windows Git Bash (recommended — installed with Git for Windows), or PowerShell, or Command Prompt
    Mac Terminal.app (built-in) or iTerm2
    Windows users: Git Bash is strongly recommended All commands in this guide use Unix-style env=dev syntax, which works in Git Bash and Mac Terminal. If you use PowerShell or Command Prompt, you must prefix commands differently — see the Windows command syntax section below.
  3. Navigate to the project folder
    cd path/to/GATestFramework
  4. Install dependencies
    npm install
    This installs all packages and downloads Playwright browsers automatically (Chrome, Firefox, Safari).
That's it! No other repos or tools are needed to run existing tests. The framework is fully self-contained.

Windows command syntax (PowerShell & Command Prompt)

Commands in this guide use env=dev prefix syntax, which works natively on Mac, Linux, and Git Bash on Windows. If you're using PowerShell or Command Prompt, use one of these alternatives:

TerminalSyntaxExample
Git Bash (Windows) / Mac Terminal env=dev command... env=dev npx playwright test tests/specFiles/ga/ --project chromium
PowerShell (Windows) $env:env="dev"; command... $env:env="dev"; npx playwright test tests/specFiles/ga/ --project chromium
Command Prompt (Windows) set env=dev && command... set env=dev && npx playwright test tests/specFiles/ga/ --project chromium
cross-env (any platform) npx cross-env env=dev command... npx cross-env env=dev npx playwright test tests/specFiles/ga/ --project chromium
Tip: cross-env works everywhere The project includes cross-env as a dependency. Prefixing any command with npx cross-env makes it work on Mac, Windows CMD, PowerShell, and Git Bash without changes.

Quick Start

Run the entire GA test suite against your dev AEM instance:

# Mac / Git Bash (Windows)
env=dev npx playwright test tests/specFiles/ga/ --project chromium

# PowerShell (Windows)
$env:env="dev"; npx playwright test tests/specFiles/ga/ --project chromium

# Any platform (using cross-env)
npx cross-env env=dev npx playwright test tests/specFiles/ga/ --project chromium

That's the only command you need to know. Everything below is for more targeted runs.

Note on command examples All remaining examples in this guide use Mac / Git Bash syntax (env=dev ...). If you're on Windows PowerShell or CMD, apply the translation from the Windows command syntax table above.

Running Tests — Advanced Options

OptionWhat it doesExample
--workers 1 Run tests one at a time (slower but more stable) env=dev npx playwright test tests/specFiles/ga/ --project chromium --workers 1
--workers 4 Run 4 tests in parallel (faster) env=dev npx playwright test tests/specFiles/ga/ --project chromium --workers 4
--retries 2 Retry failed tests up to 2 times env=dev npx playwright test tests/specFiles/ga/ --project chromium --retries 2
--headed Show the browser window (watch tests run) env=dev npx playwright test tests/specFiles/ga/ --project chromium --headed
--debug Step through tests interactively env=dev npx playwright test tests/specFiles/ga/button/ --project chromium --debug
--repeat-each 3 Run each test 3 times (flakiness check) env=dev npx playwright test tests/specFiles/ga/ --project chromium --repeat-each 3
--timeout 60000 Set test timeout to 60 seconds env=dev npx playwright test tests/specFiles/ga/ --project chromium --timeout 60000

Combining options

Options can be freely combined:

# Run button smoke tests on Chrome + Safari, headed, with retries
env=qa npx playwright test tests/specFiles/ga/button/ --grep @smoke \
  --project chromium --project webkit --headed --retries 2

Reviewing Results — HTML Report

After each test run, Playwright generates an interactive HTML report. Open it with:

npx playwright show-report playwright-report/<date>/run-<timestamp>

Or find the latest report:

# Open the most recent report in your browser
npx playwright show-report

What the report shows

  • Pass / Fail / Skip counts with percentages
  • Each test with its name, duration, tags, and status
  • Error messages and stack traces for failures
  • Screenshots captured at the point of failure
  • Videos of the full test execution (on failure)
  • Trace files for step-by-step debugging

Reviewing Results — Screenshots & Videos

ArtifactWhen capturedWhere to find
Screenshots On test failure Embedded in HTML report — click the failed test
Videos On test failure (retained) Attached to test in HTML report
Trace files On first retry Open with: npx playwright show-trace <trace.zip>
Trace Viewer Trace files let you step through a failed test action-by-action, seeing the DOM, network requests, and console logs at each step. This is the most powerful debugging tool available.

Reviewing Results — Run Logs

Every test run saves a JSON log to logs/YYYY-MM-DD/run-<timestamp>.json. These contain:

  • Test names, durations, and pass/fail status
  • Environment details (URL, browser, viewport)
  • Console errors captured from the browser

JSON results are also saved to playwright-report/<date>/run-<timestamp>/results.json for programmatic analysis.

Reviewing Results — Coverage Matrix

The file tests/data/coverage-matrix.json tracks which components have tests and in which categories. It's auto-updated by the generators. Open it to see:

  • Which components have test coverage
  • How many tests per category (smoke, regression, a11y, visual, etc.)
  • Which spec files contain those tests
  • When coverage was last updated

Reference — Components

Each component has a dedicated test suite in tests/specFiles/ga/<component>/ with up to 5 spec files:

Spec file suffixWhat it coversTags
.author.spec.ts Core tests: rendering, navigation, responsive, accessibility, negative cases @smoke @regression @a11y
.interaction.spec.ts How the component adapts when nested inside other components @interaction @regression
.matrix.spec.ts Every combination of style variants, themes, and backgrounds @matrix @regression
.visual.spec.ts Pixel comparison against Figma design baselines @visual
.images.spec.ts Broken images, missing alt text, oversized images, layout shift @regression

Plus two cross-component specs:

  • api-mock.spec.ts — Tests component behavior when APIs return errors or empty data
  • content-driven.spec.ts — Validates JCR content XML structure matches what AEM renders

Maintenance — Cleanup

Over time, logs and reports accumulate. Clean them up periodically:

WhatLocationCleanup command

HTML reports playwright-report/ Delete old date folders manually
Test videos test-results/ Delete folder contents manually
Automatic cleanup The cleanup:logs command deletes log files older than 7 days.

Maintenance — Troubleshooting

ProblemSolution
"Cannot find module" errors Run npm install again
"Browser not found" errors Run npx playwright install
Tests time out Check that the target AEM instance is running and accessible. Try --timeout 120000 for slower environments.
Auth failures Verify AEM_AUTHOR_USERNAME and AEM_AUTHOR_PASSWORD in your .env.<name> file
Tests pass locally but fail in CI Check environment variables in bitbucket-pipelines.yml and ensure the AEM instance is reachable from CI
"Fixture sync warning" The AEM style guide changed since tests were generated. Tests still run. Re-generate with the generators if needed.
Want to see the browser while tests run Add --headed to the command
Want to debug a specific test Add --debug to step through interactively

Test Generators (Advanced)

These scripts scan a live AEM instance and generate test files automatically. You only need these when adding test coverage for new components.

GeneratorWhat it doesCommand
Components Scans AEM DOM → creates Page Objects + test specs env=local npx playwright test tests/generators/generate-components.ts --project chromium --workers 1
Advanced Creates interaction, matrix, visual, and image tests env=local npx playwright test tests/generators/generate-advanced.ts --project chromium --workers 1
From CSV Reads a CSV spreadsheet of test cases → generates specs CSV_PATH=file.csv env=local npx playwright test tests/generators/generate-from-csv.ts --project chromium
From Jira Reads Jira ticket requirements → generates specs JIRA_JSON=req.json env=local npx playwright test tests/generators/generate-from-jira.ts --project chromium
HTML Summaries Generates readable HTML summaries of all test suites npx playwright test tests/generators/generate-html-summaries.ts --project chromium --workers 1
Generators require a live AEM instance The component and advanced generators connect to AEM Author (localhost:4502 by default) to scan the actual DOM. They will fail if AEM is not running.

Playwright Agent (AI-Powered)

The playwright-agent is a separate Claude Code agent that uses AI to generate tests. It lives in a sibling directory (../playwright-agent/) and writes generated code into this repo.

Available commands (run from playwright-agent directory in Claude Code)

CommandWhat it does
/automate excel <path>Generate tests from a CSV/Excel spreadsheet
/automate jira <ticket-key>Generate tests from a Jira ticket (+ optional Figma)
/automate component <name>Generate tests by scanning live AEM DOM or source code

Generated files land in:

  • Page Objects → tests/pages/ga/components/
  • Test Specs → tests/specFiles/ga/
  • Locator Sidecars → *.locators.json alongside POMs

CI/CD (Bitbucket Pipelines)

Tests run automatically on every push via Bitbucket Pipelines in two parallel steps:

StepWhat runsBrowsersWorkers
Mobile --grep @mobile Mobile-Chrome, Mobile-WebKit 4
Desktop All non-mobile tests chromium, firefox, webkit 4

Results are posted to Microsoft Teams automatically via send-report.js.
